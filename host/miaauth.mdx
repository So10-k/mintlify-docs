**MiaAuth** is the brand name for the tournament's sign-in flow. Under the hood it's **two providers running in parallel** through [Auth.js](https://authjs.dev) (the package formerly known as NextAuth):

1. **Auth0 OIDC** — the primary, customer-facing flow. The "Sign in with MiaAuth" button on `/signin` redirects to Auth0's hosted page, which sends a one-time **code** to the user's email. Branded externally as "MiaAuth, powered by Auth0 by Okta."
2. **Email magic-link** — the legacy flow, kept running for two reasons: (a) `/join` still uses it so new players get pre-created with their chosen display name, and (b) `/signin?legacy=1` exposes it as a break-glass option if Auth0 is down.

Both providers write to the same `users` table through the Drizzle Auth.js adapter, and both produce **database-backed sessions** — there are no JWTs in play. A user who first signs up via the magic link can later sign in via Auth0 (and vice versa) and end up with the same account, thanks to `allowDangerousEmailAccountLinking: true` on the Auth0 provider. The shared join key is the verified email address.

<Note>
  As a host, you don't need to wire Auth0 yourself — it's pre-configured in the deployment. This page exists so you can confidently explain the flow, troubleshoot delivery problems, and understand the trade-offs the dual-provider setup makes.
</Note>

## What each provider does

### Auth0 provider

Configured in `auth.ts` only when `AUTH_AUTH0_ID`, `AUTH_AUTH0_SECRET`, and `AUTH_AUTH0_ISSUER` are all set. The Auth0 tenant is configured for **passwordless email** delivery — Auth0 sends the user a one-time code (not a clickable link) and verifies it on their hosted page.

The provider is constructed with `allowDangerousEmailAccountLinking: true`. That flag tells Auth.js to **link the Auth0 identity to any existing tournament user with the same email**, instead of refusing to sign in or creating a duplicate `users` row. We accept the "dangerous" label because Auth0's passwordless flow verifies the email end-to-end — the user proved they can read mail at that address before Auth0 issued the token.

### Email magic-link provider

A custom Auth.js `email` provider with a 24-hour `maxAge` and a 32-character hex token generator. The verification email is sent via `sendOne` (in `lib/email-provider.ts`) which routes through whichever email provider (Resend or Brevo) the host has selected. The subject is hard-coded to **"Your magic link"** and the body uses the picture-book stationery.

Auth.js's email provider doesn't require a name — only the email. The `/join` page therefore pre-creates the `users` row with the chosen display name **before** calling `signIn("email", ...)`, so the magic-link callback finds an existing user instead of creating a nameless one.

## What MiaAuth adds on top

The shared "MiaAuth" branding wraps both providers with tournament-specific behavior:

- **Sign-in page UI** — `/signin` shows the MiaAuth lockup (sun icon, "Welcome back!" headline, three trust badges). The legacy form is hidden unless the visitor adds `?legacy=1`.
- **Display-name capture on /join** — Auth0's flow doesn't ask for a name, so first-time players go through `/join` (magic link), which collects a name and pre-creates the user. After their first sign-in, they can use either flow.
- **Registration-closed gate** — Auth.js calls a `signIn` callback before completing OIDC sign-in. The callback rejects brand-new emails if the most recent tournament has `registrationOpen = false`, and Auth.js redirects them to `/signin?error=AccessDenied`, which renders as the friendly "Registration's closed for now" banner. For the magic-link provider, the equivalent gate lives in `app/join/actions.ts` and `app/signin/actions.ts` — they bounce unknown emails to `/join`.
- **Author-role auto-promotion** — On every `createUser` event, Auth.js checks if the new user's email equals `AUTHOR_EMAIL`. If yes, that user's `role` is set to `author` immediately, which unlocks `/host` on first sign-in.
- **Branded magic-link email** — The legacy flow uses the same picture-book HTML stationery as the rest of the site, regardless of the active site theme. The subject stays "**Your magic link**" so existing recipients don't see it land in a different thread.

## What players see

Every sign-in goes through exactly two screens:

| Door | Step 1 | Step 2 |
|---|---|---|
| `/signin` (Auth0) | Click **Sign in with MiaAuth** | Enter the 6-digit code on Auth0's page |
| `/join` (magic link) | Fill in name + email, click **Send me a magic link** | Click **Play now →** in the email |
| `/signin?legacy=1` | Type email, click **Email me a magic link (legacy)** | Click **Play now →** in the email |

After step 2, the user is on `/play`, signed in. Auth0-flow users see an Auth0-hosted page briefly between step 1 and step 2; magic-link-flow users go through `/join/check-inbox` (the verify-request landing page).

<Tip>
  If a player asks "why didn't I get a password?" — there's nothing to set. MiaAuth's whole point is no passwords ever. The tournament site doesn't have a password field anywhere.
</Tip>

## Sessions and account linking

| Concern | How it's handled |
|---|---|
| **Strategy** | `session: { strategy: "database" }` — every session is a row in the `sessions` table. No JWTs. |
| **Linking** | `allowDangerousEmailAccountLinking: true` on the Auth0 provider — same email always lands on the same `users.id`. |
| **Display name** | Captured at `/join`; never changed by Auth0. If a host renames a player, the change is in `users.name` and is reflected everywhere. |
| **Role** | Stored on the `users` row. Promoted to `author` automatically when the email matches `AUTHOR_EMAIL`; promotable manually via the host **Players** section for cohosts. |
| **Sign-out** | Both flows share the same sign-out action in `app/actions/signout.ts`. Clearing the session row signs the user out everywhere. |

## Troubleshooting MiaAuth issues

| Symptom | Likely cause | What to do |
|---|---|---|
| "I never got the code/link" | Mail provider delay, spam folder, or the player's mail server blocking the sender. | Have them check spam first. Then watch `/host/email-analytics` for the message; if it's marked sent, the issue is downstream of you. |
| "The code didn't work" / "Expired" | Auth0 codes expire after a short window. | They can ask Auth0 for a new code from the same page — no need to start over from `/signin`. |
| "The link says it's expired" | Magic links expire after 24 hours or after one click. | Send them back to `/join` (new players) or `/signin` (returning) to request a new link. |
| "I'm signed in as the wrong account" | They previously signed in with a different email on this browser. | Click their name → sign out, then request a new MiaAuth code with the correct email. |
| "Auth0 is down" | Rare but possible. | Tell the player to add `?legacy=1` to the `/signin` URL and use the magic-link form below the MiaAuth button. |
| "I changed my email — can I keep my account?" | Auth.js keys identity to email; the tournament doesn't auto-migrate. | They sign up fresh; you merge their progress manually via **Players** if needed. |
| Brand-new player blocked with "Registration's closed" | The `signIn` callback or the `/join` action gate is active because the host has registration closed. | If you want to let them in, **Open sign-ups** on `/host` and have them try again. |

<Warning>
  If multiple players from the same domain report missing emails, it's almost always a delivery problem at their mail server (greylisting, aggressive filtering) — not an Auth0 outage. Check `/host/email-analytics` to confirm the message was accepted by the provider, and consider switching email provider on `/host` if the issue is widespread.
</Warning>

## What MiaAuth deliberately doesn't do

- **No passwords.** Nothing to reset, leak, or remember beyond an email address.
- **No social sign-in.** Auth0 supports Google/Apple/etc., but the tournament tenant is configured for email-only.
- **No SSO or enterprise connections.** This is a kids' quiz tournament, not an enterprise app.
- **No account deletion self-service.** Use the **remove** action in the host **Players** section (see [Managing Players](/host/managing-players)).
- **No host bypass.** Hosts sign in with MiaAuth the same way players do. The **author** role is attached to the tournament `users` row, not the Auth0 identity — so signing in with a different email puts you in player mode.

## Required environment variables

The deployment expects these env vars; the dual-provider setup is conditional on them:

| Var | Used by | When required |
|---|---|---|
| `AUTH_SECRET` | Auth.js | Always — session signing key. |
| `AUTH_AUTH0_ID` | Auth0 provider | When Auth0 is enabled (all three Auth0 vars must be set together). |
| `AUTH_AUTH0_SECRET` | Auth0 provider | As above. |
| `AUTH_AUTH0_ISSUER` | Auth0 provider | As above. The full Auth0 tenant URL, e.g. `https://your-tenant.us.auth0.com`. |
| `AUTHOR_EMAIL` | createUser event | Optional but recommended — the email that gets auto-promoted to author role on first sign-in. |
| `RESEND_API_KEY` *or* `BREVO_API_KEY` | Magic-link provider, all transactional email | At least one must be set in production. With neither, the magic-link flow falls back to logging links to the server console (dev mode). |
| `EMAIL_FROM` | Magic-link provider | Optional — defaults to `onboarding@resend.dev`. Override with your verified sender. |
